# SETUNI_Homework
homework 1

Суть:  перетворюємо сирий, дубльований набір даних (CSV ~2GB) у чітку реляційну схему. 
Mета: прибрати аномалії, зекономити місце на диску та зробити розрахунок CTR миттєвим.

1. Архітектура бази (DDL)
Використано схему «снов-флейк». Використовуємо BINARY(16) для ID замість довгих рядків, повинно добавити швидкості на індексах.

Таблиця	Опис	Чому саме так?

advertisers	Словник брендів	 : Уникаємо дублювання назв. Один раз зберегли — тисячу разів використали ID.
users	Профілі юзерів	: Відокремлюємо демографію від подій для швидкої фільтрації.
campaigns	Метадані кампаній : 	Поєднує рекламодавця з конкретним медіа-планом.
impressions	Таблиця фактів :	 impression_id у BINARY(16) — для продуктивності.
clicks	Події кліків :	Виношу окремо (1:0/1). Не кожна імпа заканчується кліком, тому не забиваємо основну таблицю порожніми полями.

2. Процес ETL (Python)

Скрипт etl_process.py працює :

Chunking: Читаємо гігантські CSV частинами (batch size: 5000).

Staging: Замість складних Join-ів у пам'яті Python, ми заливаємо все в TEMPORARY TABLE у MySQL і робимо фінальний мерж силами бази. 
Data Cleaning: Автоматична заміна NaN на NULL та конвертація таймстемпів "на льоту".



3. Як це запустити

Все загорнуто в Docker

Піднімаємо базу (в терміналі):

docker-compose up -d

Накочуємо схему:


docker exec -i adtech_mysql mysql -u root -proot_password adtech_db < schema.sql


Запускаємо ETL:

python etl_process.py

4. Аналітика та Performance
Завдяки  індексам, ці запити не "кладуть" базу :

CTR (Click-Through Rate): Розраховується через LEFT JOIN між impressions і clicks по бінарному ключу.

Advertiser Spend: Пряма агрегація cost_paid з групуванням по advertiser_id.
